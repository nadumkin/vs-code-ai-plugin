# Обзор технологий, приёмов и принципов

Документ описывает основные технологии, инженерные приёмы и архитектурные принципы,
применённые в двух частях программного комплекса: плагине-ассистенте для VS Code
(`vs-ext-ai-plugin/`) и асинхронной серверной части (`server/`).

---

## 1. Плагин VS Code

### 1.1. Технологический стек

| Технология | Применение |
|---|---|
| VS Code Extension API | точки расширения: webview-панель, команды, конфигурация |
| JavaScript (Node.js, extension host) | логика агента, транспорт, работа с редактором |
| Webview API + `postMessage` | изолированный UI чата (HTML/CSS/JS) с двусторонним обменом сообщениями |
| `SecretStorage` | защищённое хранение токена доступа (не попадает в settings.json) |
| `ws` (WebSocket-клиент) | получение результата от серверной части |
| глобальный `fetch` | HTTP-запросы к Proxy Service |

### 1.2. Ключевые приёмы

**Агентный цикл (agent loop) с вызовом инструментов.** Ответ модели интерпретируется
не как финальный текст, а как возможный набор `tool_calls` (OpenAI function calling).
`AgentRuntime` исполняет вызовы через `ToolExecutor` (чтение/замена файла, список
файлов и т.д.), добавляет результаты в историю сообщений и повторяет обращение к
модели — до тех пор, пока модель не вернёт текстовый ответ без вызовов. Каждая
итерация цикла — отдельный запрос к серверу со своим `requestId`.

**Единый формат обмена (OpenAI chat.completion).** Транспорт был заменён с прямого
синхронного вызова OpenRouter на асинхронный конвейер, но формат запроса и ответа
сохранён. Благодаря этому агентный цикл, исполнение инструментов и UI не потребовали
изменений — поменялся только клиентский класс транспорта.

**Двухфазный асинхронный транспорт.** `POST /v1/requests` возвращает `requestId`
немедленно; результат доставляется по WebSocket `ws://…/ws/{requestId}`. Таймауты
разделены: короткий на постановку в очередь (сеть/сервис), длинный (до 10 минут) на
ожидание инференса — локальные модели могут отвечать минутами. Защита от двойного
завершения промиса реализована через идемпотентный `finish()`.

**Классификация запроса (`RequestClassifier`).** Намерение пользователя (объяснение,
вопрос, правка кода) определяется до обращения к модели и управляет поведением
пост-обработки: для не-редактирующих намерений отключается применение кода из ответа.

**Fallback для слабых моделей (code-block → правка файла).** Малые локальные модели
ненадёжно следуют протоколу tool-calling, но стабильно возвращают код в fenced-блоке.
Если ответ не содержит `tool_calls`, но содержит блок кода, а намерение —
редактирование, плагин синтезирует вызов `replace_active_file` из крупнейшего блока
и проводит его через штатный поток предпросмотра diff и подтверждения. Сервер
дополняет этот приём параметром `supports_tools: false` — инструменты вообще не
объявляются слабой модели.

**Динамический выбор модели.** Список моделей запрашивается у сервера
(`GET /v1/models`) и отображается в настройках выпадающим списком; источником истины
является серверный реестр, поэтому состав моделей меняется без обновления плагина.

### 1.3. RAG: память о прошлых изменениях (`src/memory/`)

В плагине реализована подсистема retrieval-augmented generation, но её корпус — не
документация, а **собственный опыт агента**: прошлые правки кода и результаты
последовавших за ними прогонов тестов. Цель — при новой правке предупредить модель
о том, какие падения тестов вызывали лексически и структурно похожие изменения.

**Накопление знаний (индексация).** Цикл «правка → тесты» перехватывается в двух
точках `ToolExecutor`:

1. *Применение правки* (`recordPendingApply`): из diff-а строится поисковый запрос —
   `DiffNormalizer` извлекает изменённые файлы, токены (с фильтрацией стоп-слов,
   с учётом синтаксиса JVM- и C-подобных языков) и имена методов. Запрос попадает в
   очередь ожидающих (`pendingQueries`, максимум 5).
2. *Запуск тестов* (`maybeRecordTestOutput`): если команда или её вывод распознаны
   как тестовые, `StackTraceParser` извлекает из вывода упавшие тесты — тип
   исключения, сообщение, кадры стека. Все ожидающие запросы материализуются в
   записи хранилища (`MemoryStore`, персистентный JSON): «признаки diff-а →
   исход тестов». Размеры полей ограничены (600 токенов, 60 методов, 4 КБ diff).

**Обогащение для Java (`JavaEnricher`, `BytecodeFingerprint`).** Помимо лексики,
запись дополняется структурными признаками: полные имена и сигнатуры затронутых
методов, граф вызовов (callers/callees) и n-граммы байткода затронутых методов.
Это позволяет находить похожие правки даже при разной лексике (переименования,
рефакторинги), по совпадению структуры.

**Поиск (retrieval) — двухступенчатый.** При каждом следующем применении правки:

1. *Кандидаты*: классический **BM25** (k1 = 1.5, b = 0.75) по токенам diff-а против
   всех сохранённых записей — собственная реализация без внешних зависимостей
   (`BM25Index`), корпус мал и целиком в памяти.
2. *Реранкинг* (`Reranker`): топ-N кандидатов (по умолчанию 50) переупорядочиваются
   взвешенной суммой четырёх компонент — нормированный BM25 + сходство Жаккара по
   множествам методов, рёбер графа вызовов и байткод-n-грамм. Веса компонент
   настраиваются (`memory.rerank.weights.*`), финально отбирается top-K
   (по умолчанию 3, `memory.topK`).

Выбор BM25 + структурный реранкинг вместо векторных эмбеддингов сознательный:
корпус — десятки записей, запрос — код (где лексическое совпадение
идентификаторов информативнее семантической близости), и всё работает офлайн в
extension host без модели эмбеддингов и векторной БД.

**Аугментация (augmentation).** Найденные записи агрегируются
(`aggregatePredictions`): подсчитываются частоты типов исключений и кадры стека,
повторяющиеся в большинстве совпадений. `buildPredictionPromptText` формирует
текстовый блок «[Memory of past similar changes]» — агрегированный сигнал плюс
top-K прецедентов с их score (с разложением по компонентам) и краткими падениями —
и он внедряется в следующий промпт модели. Модель получает шанс упреждающе
исправить код до запуска тестов.

**Жизненный цикл и управление.** Подсистема отключаема (`memory.enabled`),
хранилище инвалидируется явно; записи привязаны к рабочей области. Ошибки любого
этапа (парсинг Java, байткод, разбор вывода) деградируют мягко — теряется
обогащение, но не работоспособность.

### 1.4. Принципы

- **Минимальная связность UI и логики:** webview общается с extension host только
  сообщениями; состояние (`postState`) — единственный канал синхронизации.
- **Деградация вместо отказа:** слабая модель без tool-calling всё равно приводит к
  применимой правке через fallback.
- **Секреты отдельно от настроек:** токен — в `SecretStorage`, адреса и модель — в
  обычной конфигурации.

## 2. Серверная часть

### 2.1. Технологический стек

| Технология | Применение |
|---|---|
| FastAPI + uvicorn | все три сервиса (Proxy, Request, LLM) |
| RabbitMQ + aio-pika | очереди `requests` и `responses` |
| PostgreSQL + SQLAlchemy (async) + asyncpg | журнал запросов, единый источник истины |
| pydantic / pydantic-settings | схемы данных и env-конфигурация |
| httpx | исходящие HTTP-вызовы (OpenRouter, llama.cpp, метаданные) |
| HuggingFace Transformers + accelerate | локальный инференс (`hf_local`) |
| bitsandbytes (NF4) | 4-битная квантизация для посадки 7B-модели в 12 GB VRAM |
| llama.cpp (`llama-server`) | запуск MoE-модели с выгрузкой экспертов в ОЗУ |
| Docker Compose (+ CUDA-override) | оркестрация; один код для CPU- и GPU-сред |

### 2.2. Архитектурные приёмы

**Асинхронный конвейер с разделением приёма и доставки.** Proxy принимает запрос и
мгновенно возвращает `requestId`; инференс выполняется фоновым потребителем очереди;
результат доставляется отдельным сервисом по WebSocket. Клиент не держит HTTP-соединение
на время инференса.

**СУБД как единственный источник истины.** Очереди переносят только `{requestId,
status}` — полезная нагрузка всегда читается из PostgreSQL. Сообщения брокера
становятся идемпотентными уведомлениями: потерянное или дублированное сообщение не
ломает доставку.

**Устранение состояния гонки при доставке.** Request Service регистрирует
`asyncio.Event` **до** чтения БД, а LLM Service публикует уведомление **только после**
коммита результата. Два пути доставки (connect-before — ожидание события,
connect-after — немедленное чтение готового ответа) используют один и тот же код
«прочитай БД → отправь».

**Паттерн «Адаптер» для моделей.** Единый интерфейс
`BaseModelAdapter.generate(ChatRequest) → chat.completion` скрывает четыре бэкенда:
`echo` (офлайн-заглушка), `openrouter` (любой OpenAI-совместимый API), `llama_server`
(локальный llama.cpp-сайдкар), `hf_local` (Transformers). Состав моделей задаётся
декларативно в JSON; реестр пропускает записи с отсутствующими зависимостями
(например, `hf_local` без torch на CPU-образе) — набор моделей определяется средой.

**Аутентификация через общий файл с hot-reload.** Токены — в JSON-файле, смонтированном
в Proxy и Request Service; файл перечитывается по изменению mtime, что позволяет
выдавать/отзывать токены без перезапуска.

### 2.3. Приёмы оптимизации локального инференса

**Автоопределение устройства (CUDA → MPS → CPU)** с выбором типа данных: `bfloat16`
на CUDA (численно стабильнее `float16` на современных GPU), `float16` на MPS,
`float32` на CPU. Устройство протоколируется при старте и в `/health`.

**Квантизация под объём VRAM.** 7B-модель в bf16 (~15 GB) не помещается в 12 GB —
`device_map="auto"` начал бы выгружать слои в ОЗУ с падением скорости на порядок.
NF4-квантизация (bitsandbytes, double quant, compute в bf16) сжимает её до ~5 GB —
модель целиком остаётся на GPU. Модели, помещающиеся в VRAM (3B), работают без
квантизации.

**Fused-ядра внимания.** Используется `flash_attention_2` при наличии пакета, иначе —
встроенный в PyTorch SDPA (те же FlashAttention-идеи без компиляции из исходников,
для которой в runtime-образе нет nvcc).

**Ленивая загрузка и политика «одна модель в VRAM».** Веса загружаются при первом
запросе к конкретной модели; при переключении на другую предыдущая выгружается
(`gc.collect()` + `torch.cuda.empty_cache()`). Глобальный lock сериализует
«загрузку + генерацию», исключая вытеснение весов посреди декодирования.

**MoE с частичной выгрузкой экспертов (llama.cpp).** Qwen3-30B-A3B (30.5B параметров,
~3.3B активных на токен) запускается через `llama-server`: attention и KV-кэш — на
GPU, эксперты делятся между GPU и ОЗУ (`-ngl 99 --n-cpu-moe N`). Приём опирается на
свойство MoE: на каждом токене используется малая доля весов, поэтому «холодные»
эксперты можно держать в медленной памяти. Дополнительно: квантованный KV-кэш
(`-ctk/-ctv q8_0`) для экономии VRAM на длинном контексте (32k), увеличенные батчи
(`-ub/-b 2048`) для скорости обработки промпта, `--jinja` для родного чат-шаблона
и нативного tool-calling.

**GGUF-квантизация.** Веса MoE — в формате GGUF (Q4_K_XL, ~17.7 GB вместо ~61 GB в
fp16); загрузка с HuggingFace при первом старте в постоянный том.

### 2.4. Принципы

- **Расширяемость без изменения протокола:** новая модель = запись в JSON-реестре;
  новый бэкенд = один класс-адаптер.
- **Переносимость:** один код и один compose-файл; GPU-специфика изолирована в
  override-файле (`docker-compose.cuda.yml` + `Dockerfile.cuda`).
- **Конфигурация через окружение** (12-factor): все параметры — env-переменные с
  разумными значениями по умолчанию; секреты (ключ OpenRouter) не хранятся в коде.
- **Наблюдаемость:** `/health` у каждого сервиса, healthcheck-и в compose,
  журналирование каждого запроса в СУБД со статусами `queued → processing →
  done/error`, тексты ошибок апстрима прокидываются до клиента.
- **Отказоустойчивость по частям:** сбой одной модели/адаптера не валит сервис;
  недоступность зависимостей понижает функциональность (реестр пропускает модель),
  а не останавливает запуск.

## 3. Сквозные решения

- **Согласованные таймауты по всей цепочке:** ожидание WebSocket на сервере и в
  плагине — 600 с; таймаут адаптера llama.cpp — 600 с; короткие таймауты только на
  быстрых операциях (постановка в очередь, список моделей).
- **Разделение ключей:** плагин знает только токен доступа к Proxy; ключ OpenRouter
  живёт исключительно на сервере и никогда не передаётся локальным бэкендам.
- **Тестируемость конвейера без IDE:** автономный `test_client.py` проверяет весь
  путь (POST → очередь → инференс → БД → WebSocket), включая сценарий позднего
  подключения (`--delay`).
