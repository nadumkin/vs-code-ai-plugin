# Серверная часть AI-ассистента: архитектура и реализация

## 1. Введение

Целью работы являлась разработка серверной части программного комплекса, обслуживающего
интеллектуальный плагин-ассистент для среды разработки VS Code. В отличие от исходной схемы, в которой
клиент обращался к языковой модели синхронно, поставлена задача построения **асинхронного бэкенда**,
разделяющего приём запроса и доставку результата, обеспечивающего журналирование обращений,
горизонтальную развязку компонентов и возможность подключения сменных моделей без изменения протокола.

## 2. Архитектура

Система реализована в виде **микросервисной архитектуры** из трёх прикладных сервисов, брокера сообщений
и реляционной СУБД, разворачиваемых единым описанием `docker-compose`:

| Компонент | Технология | Назначение |
|---|---|---|
| **Proxy Service** | FastAPI (порт 8000) | приём запроса, проверка токена, журналирование, постановка в очередь |
| **Request Service** | FastAPI + WebSocket (8090→8081) | доставка результата клиенту по идентификатору запроса |
| **LLM Service** | FastAPI + фоновый потребитель | инференс модели через адаптеры, запись результата |
| **Queue** | RabbitMQ (5672; UI 15672) | очереди `requests` и `responses` |
| **Database** | PostgreSQL (5432) | журнал запросов (запрос, статус, ответ) |

Связующий слой вынесен в общий пакет `common`: конфигурация (`pydantic-settings`), асинхронный доступ к
СУБД (`SQLAlchemy` + `asyncpg`), ORM-модель запроса, обёртка брокера (`aio-pika`), верификация токенов и
схемы данных (`pydantic`). Прикладной стек — `FastAPI`/`uvicorn`.

## 3. Конвейер обработки запроса

Обработка одного обращения организована по следующему конвейеру:

1. Плагин выполняет `POST /v1/requests` с заголовком `Authorization: Bearer <token>`.
2. **Proxy Service** проверяет токен, генерирует `requestId` (UUID), создаёт в СУБД запись со статусом
   `queued`, публикует уведомление `{requestId}` в очередь `requests` и возвращает клиенту `requestId`.
3. **LLM Service** потребляет очередь `requests`, считывает полезную нагрузку из СУБД (статус `processing`),
   по идентификатору модели выбирает адаптер, выполняет инференс, записывает результат (статус `done`/`error`)
   и публикует `{requestId, status}` в очередь `responses`.
4. Плагин открывает WebSocket-соединение `/ws/{requestId}` с **Request Service**, которое доставляет ответ.

Очереди переносят лишь идентификатор и статус; **единственным источником истины** является СУБД. Это
унифицирует пути доставки и делает сообщения брокера идемпотентными уведомлениями.

## 4. Реализация ключевых механизмов

**4.1. Асинхронная доставка и устранение состояния гонки.** Request Service поддерживает два сценария:
*connect-after* (ответ уже готов на момент подключения — извлекается из СУБД и отправляется немедленно) и
*connect-before* (ожидание сигнала из очереди `responses`). Корректность обеспечивается порядком операций:
объект синхронизации (`asyncio.Event`) регистрируется **до** чтения СУБД, а уведомление публикуется
**только после** фиксации результата в СУБД; тем самым уведомление, пришедшее в промежутке, не теряется.
Время ожидания ограничено таймаутом (`ws_response_timeout`).

**4.2. Паттерн «Адаптер» и реестр моделей.** Введён единый интерфейс `BaseModelAdapter.generate(ChatRequest)`,
возвращающий ответ в формате OpenAI `chat.completion`, что гарантирует одинаковые вход и выход независимо
от бэкенда. Состав моделей задаётся декларативно в `models.json`; реализованы адаптеры: `echo`
(офлайн-заглушка), `openrouter` (любой OpenAI-совместимый HTTP-эндпоинт), `llama_server`
(локальный `llama.cpp`-сайдкар, в т.ч. запуск MoE с выгрузкой экспертов в ОЗУ) и `hf_local` (локальный
инференс на `transformers`). Реестр инстанцирует адаптеры из конфигурации и **корректно пропускает**
записи, чьи зависимости отсутствуют в данном образе (например, `hf_local` при отсутствии `torch`), что
делает набор моделей зависящим от среды исполнения.

**4.3. Аутентификация.** Токены доступа хранятся в общем JSON-файле, монтируемом в Proxy и Request Service;
оба сервиса выполняют верификацию. Файл перечитывается при изменении времени модификации, что позволяет
обновлять список токенов без перезапуска.

**4.4. Локальный инференс.** Адаптер `hf_local` выполняет автоматическое определение устройства по
приоритету CUDA → MPS → CPU (с протоколированием при старте и в `GET /health`), выбирает тип данных
(`float16`/`float32`) и на CUDA размещает крупные модели через `accelerate` (`device_map="auto"`). Реализован
разбор вызовов инструментов: ответ модели в форматах `<tool_call>`, ограждённого JSON-блока или «голого»
JSON преобразуется в `tool_calls` с проверкой имени по списку объявленных инструментов. Для маломощных
моделей предусмотрен режим без инструментов (`supports_tools`).

**4.5. Хранение и журналирование.** Каждое обращение фиксируется в таблице `requests` (идентификатор, токен,
модель, исходный запрос, статус, ответ, отметки времени), что обеспечивает прослеживаемость и поддержку
сценария отложенного подключения клиента.

## 5. Развёртывание

Базовая конфигурация (`docker-compose.yml`) поднимает СУБД и брокер с проверками готовности
(`healthcheck`), а также прикладные сервисы; общие файлы (`tokens.json`, `models.json`) и кэш весов
(`hf_cache`) подключаются томами. Для исполнения на GPU предусмотрен слой переопределения
(`docker-compose.cuda.yml` + `Dockerfile.cuda` на базе образа PyTorch с CUDA), резервирующий устройство и
подключающий расширенный список моделей. Тем самым один и тот же код исполняется как на CPU, так и на
NVIDIA-хосте без модификаций.

## 6. Тестирование и верификация

Функционирование подтверждено сквозными проверками: прохождение конвейера для обоих сценариев доставки
(*connect-before* и *connect-after*); негативные проверки авторизации (`HTTP 401` и отклонение
WebSocket-рукопожатия); переходы статусов записи в СУБД (`queued → processing → done`); работа эндпоинта
перечня моделей `GET /v1/models`; реальный локальный инференс модели `Qwen2.5-Coder-0.5B`; корректность
логики автоопределения устройства.

## 7. Ограничения и направления развития

Маломощные локальные модели (≈0.5B) нестабильно следуют протоколу вызова инструментов, что частично
компенсировано разбором нестандартных форматов и режимом без инструментов; для надёжной агентной работы
рекомендуется применение моделей бо́льшего размера (через `openrouter`, `llama_server` или `hf_local` на
GPU). Использование MPS внутри контейнера невозможно, в связи с чем GPU-исполнение ориентировано на
NVIDIA-хост. Перспективными направлениями являются горизонтальное масштабирование LLM-сервиса (несколько
потребителей очереди), потоковая доставка ответа и расширение набора адаптеров.

## 8. Заключение

Разработан асинхронный серверный комплекс, реализующий разделение приёма запроса и доставки результата
через брокер сообщений и WebSocket, журналирование обращений в реляционной СУБД, единый интерфейс
подключения моделей по паттерну «Адаптер» и контейнеризованное развёртывание для CPU- и GPU-сред.
Архитектурные решения обеспечивают расширяемость (добавление моделей и адаптеров без изменения протокола),
отказоустойчивость доставки (СУБД как источник истины, корректная обработка состояния гонки) и
переносимость между средами исполнения.
