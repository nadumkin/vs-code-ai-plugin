# QKV-память по истории diff: реализация триады query → key → value

## Контекст и цель

В Java Agentic IDE требуется давать агенту **мгновенную обратную связь** ещё до запуска реальных тестов. Идея — хранить пары «изменение кода → результат тестов» и при появлении нового изменения находить семантически похожие исторические случаи. Если в прошлом аналогичное изменение приводило к падению `NullPointerException` в конкретном месте, агент видит это предсказание сразу после применения diff и может проактивно скорректировать код.

- **query** — diff, только что сгенерированный агентом
- **key** — нормализованное обогащённое представление прошлого diff
- **value** — результат запуска тестов для этого diff (стек-трейсы упавших тестов или маркер успеха)

## Архитектура

Весь пайплайн вынесен в отдельный модуль [`src/memory/`](../src/memory/), интегрированный в существующий agent loop через две точки врезки в `ToolExecutor`. Модуль состоит из узких компонентов, каждый отвечает за один слой:

| Файл | Слой |
|---|---|
| [`MemoryStore.js`](../src/memory/MemoryStore.js) | Персистентное хранилище JSON |
| [`DiffNormalizer.js`](../src/memory/DiffNormalizer.js) | Лексическая нормализация diff (для всех языков) |
| [`JavaSourceParser.js`](../src/memory/JavaSourceParser.js) | Парсер Java-исходников (типы, методы, enclosing) |
| [`JavaEnricher.js`](../src/memory/JavaEnricher.js) | Обогащение Java-diff: затронутые методы + workspace call graph |
| [`BytecodeFingerprint.js`](../src/memory/BytecodeFingerprint.js) | Опциональный байткод-отпечаток через `javap` |
| [`BM25Index.js`](../src/memory/BM25Index.js) | Лексический индекс BM25 |
| [`Reranker.js`](../src/memory/Reranker.js) | Реранкинг: BM25 + Jaccard по methods/calls/bytecode |
| [`StackTraceParser.js`](../src/memory/StackTraceParser.js) | Парсер вывода тестов и компилятора в структурированные failures |
| [`MemoryManager.js`](../src/memory/MemoryManager.js) | Фасад: оркестрация record / lookup / aggregate |

Поток данных:

```
Apply diff
    ↓
DiffNormalizer.buildQuery(changes) ──→ tokens, methods (heuristic), rawDiff
JavaEnricher.enrichChanges(changes) ──→ FQN methods, callers/callees
BytecodeFingerprint.build(methods) ──→ n-gram opcodes (опционально)
    ↓
MemoryManager.recordPendingApply ──→ pendingQuery в очереди
    ↓
MemoryStore.getEntries → BM25Index.search → top-N
    ↓
Reranker.rerank → top-K с компонентами score
    ↓
MemoryManager.aggregatePredictions → топ-исключения, общие frames
    ↓
buildPredictionPromptText → system-message в очередную LLM-итерацию
+ display-message в чат
─────────────────────────────────────────────
Запуск тестов через инструмент агента
    ↓
StackTraceParser.parse → структурированные failures
    ↓
MemoryManager.maybeRecordTestOutput → MemoryStore.addEntry
    ↓
Пара (key, value) сохранена; pendingQuery очищен
```

## 1. Хранилище

**Файл:** [`MemoryStore.js`](../src/memory/MemoryStore.js)

**Решение.** Простое JSON-хранилище в `<workspace>/.aiAgentAssistant/memory/store.json`. SQLite/embedding-индекс отложены — для MVP объёма лексический BM25 in-process работает быстро, а портативность JSON делает кэш переносимым между разработчиками. Путь настраивается через `aiAgentAssistant.memory.storagePath`.

**Схема записи:**

```jsonc
{
  "version": 1,
  "entries": [
    {
      "id": "rec_q_1_1715455…",
      "createdAt": "2026-05-11T22:00:00Z",
      "files": ["src/main/java/com/example/Foo.java"],
      "tokens":   [...],     // для BM25
      "methods":  [...],     // FQN сигнатуры затронутых методов
      "callGraph": { "callers": [...], "callees": [...] },
      "bytecode": { "ngrams": [...] } | null,
      "rawDiff":  "--- a/...\n+++ b/...\n- ...\n+ ...",
      "value": {
        "failures": [{ "exception", "message", "frames": [...] }],
        "success":  false,
        "commandText": "mvn test",
        "capturedAt": "2026-05-11T22:00:42Z"
      }
    }
  ]
}
```

Загрузка кэширует содержимое в памяти, `addEntry` пушит в кэш и атомарно сериализует обратно. Лимиты на размер каждого поля (`MAX_STORED_TOKENS=600`, `MAX_STORED_METHODS=60`, `MAX_STORED_CALLS=60`, `MAX_STORED_NGRAMS=400`, `MAX_RAW_DIFF_CHARS=4096`) предотвращают разрастание JSON-файла.

## 2. Нормализация diff в key

### 2.1 Лексический слой (для всех языков)

**Файл:** [`DiffNormalizer.js`](../src/memory/DiffNormalizer.js)

На вход приходит набор `{path, originalContent, proposedContent}` — это applied changes из staged diff. Алгоритм:

1. **Построение line-diff** по принципу мультимножества строк: проход по `original.split('\n')` и `proposed.split('\n')` с убыванием встречаемости в counter-Map даёт `{added, removed}` без зависимости от порядка дубликатов.
2. **Очистка комментариев** по типу файла (`//`, `/* */`, `#`).
3. **Токенизация** регулярным выражением `[A-Za-z_][\w$]*`, плюс **раскладка по camelCase / snake_case**: `getUserName` → `[getusername, get, user, name]`. Это повышает recall: правка одного метода найдёт исторический диф, где менялся метод с похожим именем.
4. **Эвристическое извлечение сигнатур** методов по регулярке для Java/Kotlin/Scala/C#, JS/TS, Python, Go. Сигнатура хранится как `name(Np)` (число параметров) — хорошее соотношение точность/устойчивость без полного типчекинга.
5. Возвращается `{files, tokens, methods, rawDiff}`.

Стоп-слова (`the, and, for, this, true, false, null, none, …`) и идентификаторы короче 2 символов отбрасываются.

### 2.2 Java AST-слой (для `.java` файлов)

**Файлы:** [`JavaSourceParser.js`](../src/memory/JavaSourceParser.js), [`JavaEnricher.js`](../src/memory/JavaEnricher.js)

**Решение.** Для Java делаем точное обогащение: enclosing-метод на изменённую строку и workspace-wide call graph. От полноценного AST через Spoon/JDT отказались в пользу собственного **brace-tracking-парсера** на JS — нет JDK-зависимости и helper-jar не нужно деплоить. Парсер покрывает 95% реальной Java: классы, интерфейсы, enum'ы, record'ы, вложенные типы, аннотации, generics, методы с throws.

**Что делает JavaSourceParser:**

1. `stripCommentsAndStrings` — заменяет содержимое строковых литералов и комментариев на пробелы, **сохраняя смещения**. Это позволяет дальше парсить по «чистому» тексту, но видеть оригинальные позиции.
2. Сканирует header'ы `class|interface|enum|record\s+Name`, через brace-counter находит `{...}` тела типа.
3. Внутри тела типа — с предварительным построением «skip-интервалов» для вложенных типов — ищет методы по паттерну «заголовок-токены, заканчивающиеся на `name(params){`», валидирует что токенизированный заголовок содержит минимум `модификатор/тип возврата + имя`, а не `if (...)` или `while (...)`.
4. Возвращает иерархию `{packageName, types: [{fqn, kind, methods: [{name, paramText, signature, fqn, headerStart, bodyStart, bodyEnd, bodyText}]}]}`.

**API:** `findEnclosingMethod(parsed, source, lineNumber)` для каждой изменённой строки находит **минимальный по длине** enclosing-метод (точное соответствие, даже если вложенный).

**Что делает JavaEnricher:**

1. Для каждого `.java`-change парсит обе версии (`original`, `proposed`).
2. Считает изменённые номера строк (через мультимножественный line-diff из DiffNormalizer-логики) для обеих версий.
3. Для каждой изменённой строки → enclosing-метод → собирает множество **затронутых методов** (FQN формата `com.example.Foo#greet(1p)`).
4. **Callees:** в теле каждого затронутого метода ищет регуляркой `\b<ident>\s*\(` — все вызываемые имена методов, фильтруя ключевые слова (`if, for, while, …`).
5. **Callers:** через `vscode.workspace.findFiles('**/*.java', exclude, 200)` сканирует workspace, для каждого файла парсит и проверяет какие методы вызывают любое из имён затронутых методов. Вернёт FQN найденных вызывающих методов.
6. Лимиты: `MAX_CALLER_FILES=200`, `MAX_CALLERS=50`, `MAX_CALLEES=50`.

### 2.3 Байткод-отпечаток (опциональный)

**Файл:** [`BytecodeFingerprint.js`](../src/memory/BytecodeFingerprint.js)

**Решение.** Заказчик предложил Левенштейн или n-граммы. Выбраны **n-граммы (n=3)** — Левенштейн на длинных байткод-строках O(N×M) даёт ту же сигнальность за большую цену; на нормализованных опкодах различия дискретны и Jaccard на n-граммах их ловит дешевле.

**Алгоритм:**

1. Режим работы — настройка `aiAgentAssistant.memory.java.bytecodeProbe`: `auto` (попытаться, если `javap` в PATH), `off`, `force` (требовать).
2. Поиск `.class`-файла для каждого затронутого FQN-типа: проход по стандартным output-папкам Maven/Gradle/IntelliJ/Eclipse (`target/classes`, `build/classes/java/main`, `out/production/classes`, `bin`, …) — настраивается через `memory.java.classOutputPaths`. Если не найдено — fallback через `vscode.workspace.findFiles('**/<ClassName>.class')`.
3. Запуск `javap -c -p -constants <class-file>`.
4. Парсинг output: для каждой `Method NAME(...)` вытаскивает opcode-последовательность (отбрасывая offsets `0:`, операнды `#7`, метки `14`, комментарии `//`).
5. **Нормализация опкодов:** `aload_0 → aload`, `iload_2 → iload` (через `replace(/_(\d+)$/, "")`). Это устойчивость к перемене порядка локальных переменных.
6. Скользящее окно n=3 → множество строк-«триграмм» вида `aload ifnonnull new`.
7. Глобальный лимит `MAX_NGRAMS_TOTAL=400`.

Если JDK не установлен или класс не скомпилирован — `bytecode = null`, реранкер просто получит нулевой Jaccard в этой компоненте.

## 3. Query — то же самое, что key

`MemoryManager.buildEnrichedQuery(changes)`:

1. Базовый лексический проход через `DiffNormalizer.buildQuery`.
2. Если включено `memory.java.enabled` — `JavaEnricher.enrichChanges` дополняет `methods`, `callers`, `callees`.
3. Если включено `memory.java.bytecodeProbe != off` и есть `javap` — `BytecodeFingerprint.build` добавляет `bytecode.ngrams`.
4. Все exception'ы внутри обогащения обёрнуты try/catch с логированием в Output — query гарантированно построится хотя бы по лексике.

Query сохраняется в очередь `pendingQueries` (FIFO, max 5) — он будет связан с результатом следующего запуска тестов.

## 4. Поиск query → key

### 4.1 Лексика BM25

**Файл:** [`BM25Index.js`](../src/memory/BM25Index.js)

Классический BM25 (`k1 = 1.5`, `b = 0.75`). Индекс строится на лету по всем entries с непустым `tokens`. Query-терms из `query.tokens` приводятся к lowercase, дедуплицируются. Для каждого документа считается:

```
score = Σ over query terms:  IDF(term) × (tf × (k1+1)) / (tf + k1 × (1 − b + b × docLen/avgDocLen))
IDF(term) = ln(1 + (N − df + 0.5) / (df + 0.5))
```

Возвращаются документы со score > 0, отсортированные по убыванию.

**Решение.** Не используется внешний движок (lunr, elasticlunr) — для in-process поиска по сотням-тысячам entries собственный BM25 даёт минимум зависимостей и достаточную скорость (O(N×Q)).

### 4.2 Реранкинг

**Файл:** [`Reranker.js`](../src/memory/Reranker.js)

Из BM25 берётся **top-N** (по умолчанию 50, настраивается `memory.rerank.topN`). Для каждого кандидата считаются три Jaccard-сходства:

| Метрика | Формула |
|---|---|
| `simMethods` | `\|query.methods ∩ key.methods\| / \|query.methods ∪ key.methods\|` |
| `simCalls` | `\|(query.callers∪callees) ∩ (key.callers∪callees)\| / \|union\|` |
| `simBytecode` | `\|query.bytecode.ngrams ∩ key.bytecode.ngrams\| / \|union\|` |

**Решение.** Для callers/callees заказчик допускал и косинус по бинарным векторам — на множестве это эквивалент Jaccard, но Jaccard проще считать без построения векторов. На байткоде Jaccard по 3-граммам уже учитывает локальные перестановки.

**Финальный score:**

```
final = w_bm25 × bm25_norm
      + w_methods × simMethods
      + w_calls × simCalls
      + w_bytecode × simBytecode

bm25_norm = bm25_score / max_bm25_in_topN   ∈ [0, 1]
```

Веса настраиваются: `memory.rerank.weights.{bm25, methods, calls, bytecode}`. Дефолты `1.0 / 1.0 / 0.7 / 0.5` подобраны эмпирически — методы и токены — самый сильный сигнал, callers/callees вторичны (менее точные при текстовом scan), байткод — самый шумный (зависит от сборки).

Возвращаются top-K (по умолчанию 3, настраивается `memory.topK`) с разбивкой по компонентам — для трассировки и дальнейшего тюнинга весов.

## 5. Извлечение value + агрегация top-K

**Файл:** [`MemoryManager.js`](../src/memory/MemoryManager.js)

После реранкинга есть `predictions: [{score, components, files, value, …}]`. Для каждого предсказания из `value.failures` извлекается список `{exception, message, frames: [{class, method, location}]}` — они показываются «как есть» в Memory-блоке.

### Агрегация

Заказчик попросил отдельно: «выделить наиболее часто встречающийся тип исключения или общие строки в стеке». Реализовано в `aggregatePredictions(predictions)`:

```
input: top-K предсказаний
output: {
  sampleSize:     K,
  totalFailures:  суммарное число failures по top-K,
  successCount:   сколько из K — это «тесты прошли»,
  topExceptions:  до 3 типов exceptions, отсортированных по частоте,
  commonFrames:   до 5 фреймов вида class.method(file:line),
                  встречающихся в ≥ ⌈K/2⌉ совпадениях
}
```

Порог `⌈K/2⌉` гарантирует «commonFrames» — это действительно общие фреймы большинства, а не уникальные хвосты одного предсказания.

Агрегация попадает в **два места** одновременно:

1. **system-сообщение для LLM** (`buildPredictionPromptText`) — сводка идёт **первой**, потом per-prediction детали:
   ```
   [Memory of past similar changes]
   Aggregated signal (top-K=3): failed=4 across 3 prior diff(s), success=0.
   - Most frequent exception types: NullPointerException ×2, IllegalArgumentException ×1.
   - Frames recurring in ≥2/3 matches:
       com.example.Foo.greet(Foo.java:42) ×2
   ```
2. **UI-блок в чате** (фиолетовый прямоугольник «Сводка top-K»):
   - топ-исключения списком
   - общие фреймы моноширинным шрифтом
   - под каждым предсказанием — компонентная разбивка `bm25 1.0 • methods 0.5 • calls 0.7 • bytecode 0`

## 6. Интеграция в agent loop

Две точки врезки в [`ToolExecutor.js`](../src/agent/ToolExecutor.js):

### 6.1 Точка записи query — после Apply

В `applyPendingChanges` (когда пользователь подтвердил staged diff) вызывается `recordAppliedChanges(session.changes)`:

1. `MemoryManager.recordPendingApply` строит обогащённый query.
2. Делает lookup в существующих записях → BM25 → реранкер → top-K predictions.
3. Возвращает `{queryId, predictions, promptHint, displayMessage}`.
4. `displayMessage` уходит в чат (`role: "memory"`).
5. `promptHint` записывается как **новое system-message** в `continuationState.messages`, которое уходит модели на следующую итерацию.

Если режим `aiAgentAssistant.execution.autoApplyFileChanges = true`, та же логика срабатывает прямо в `handleFileMutation`, и `memoryPromptHint` попадает в JSON tool-результата — модель видит подсказку нативно.

### 6.2 Точка записи value — после теста

В `runShellCommand` и `runBashScript` после завершения процесса вызывается `noteTestExecution({commandText, stdout, stderr})`:

1. `StackTraceParser.isTestCommand(commandText)` — регулярка для `mvn test`, `gradle test/check`, `pytest`, `jest`, `cargo test`, `dotnet test`, `npm/yarn/pnpm test|run build`, `javac`, `tsc`, `rspec`.
2. `StackTraceParser.outputLooksLikeTests(text)` — fallback по маркерам: `Tests run:`, `BUILD FAILURE`, `BUILD SUCCESS`, `=== FAILURES ===`, `[ERROR]`, `JUnit Jupiter`, `COMPILATION ERROR` и пр.
3. Если хоть один из критериев сработал и `pendingQueries` непуст — парсит вывод:
   - JVM-стек (`at <fqn>.<method>(<file>:<line>)`)
   - Exception-строки (`java.lang.NullPointerException: …`)
   - pytest (`E   AssertionError: …`)
   - **Maven-компилятор** (`[ERROR] /path/File.java:[12,22] message`) → синтетический `java.lang.compile.CompilationError` с фреймами `<compile>`
   - **javac** (`/path/File.java:12: error: …`)
   - Если ни одного, но в выводе есть `BUILD FAILURE` — синтетический `BuildFailure`
4. Все накопленные `pendingQueries` слипаются с этим value и идут в `MemoryStore.addEntry` → `save()`.
5. В чат уходит system-сообщение `Memory: сохранены стек-трейсы (<exception> ×N) для K прошлых diff(ов)`.

System-prompt агента ([`AgentRuntime.js`](../src/agent/AgentRuntime.js)) дополнен инструкцией: «After applying a change you may receive a system or tool message starting with `[Memory of past similar changes]`. Treat that block as predicted failures from similar past diffs: read the stack traces, decide whether the current change has the same risk, and proactively patch the issue before running real tests».

## 8. Соответствие требованиям заказчика

| Требование из ТЗ | Как реализовано |
|---|---|
| Хранилище пар `(key, value)` | `MemoryStore` — JSON в workspace |
| key = токенизированный текст без шума | `DiffNormalizer.tokenize` + camelCase split + стоп-слова |
| key = множество затронутых методов с сигнатурами | `JavaEnricher.affectedMethods` (FQN + paramCount) |
| key = контекст вызовов из статического графа | `JavaEnricher.findCallersInWorkspace` + scan callees в теле метода |
| key = байткод changed methods | `BytecodeFingerprint` (`javap` → нормализованные n-граммы) |
| query = тот же путь обогащения | `MemoryManager.buildEnrichedQuery` |
| Лексика BM25 → top-N | `BM25Index.search` + `Reranker.topN` |
| Уточнение Jaccard по методам | `Reranker.simMethods` |
| Уточнение Jaccard / cosine по callers∪callees | `Reranker.simCalls` (Jaccard, эквивалент косинуса на бинарных множествах) |
| Уточнение Левенштейн / n-gram по байткоду | `Reranker.simBytecode` (n-gram, n=3) |
| Взвешенная сумма со настраиваемыми весами | `Reranker.rerank` + 4 weight-настройки |
| Извлечение value (топ-K stack traces) | `MemoryManager.recordPendingApply` возвращает predictions с failures |
| Агрегация: топ-исключения, общие frames | `MemoryManager.aggregatePredictions` |
| Возврат предсказания агенту | system-message + Memory-блок в чате |

## 9. Ограничения и явные tradeoffs

- **Парсер Java — собственный brace-tracker, не AST.** Покрывает обычный код, может давать ложно-положительные сигнатуры на необычных конструкциях (например, лямбды на одной строке с return-типом `Function<X, Y>` без модификатора). Принято осознанно ради нулевых внешних зависимостей.
- **Call graph — текстовый и без разрешения перегрузок.** Если в workspace есть два метода с именем `process(...)`, oба попадут в callers без различения. Для целевой задачи (грубый сигнал семантической близости) этого достаточно.
- **Байткод — опционален и нормализован грубо.** Замены `aload_N → aload` и пропуск операндов делают отпечаток устойчивым к перенумерациям локалов, но теряет информацию о константах. Это сознательный выбор: точные операнды дали бы переобучение на несущественные различия.
- **Запись пары (key, value) сейчас — batched.** Один Apply = одна запись на весь батч. Гранулярность по файлу — одно из направлений развития.
- **Веса реранкера — эмпирические.** Заказчик допускал «обучить на истории» — для этого нужен eval-harness с аннотированным датасетом, который не входил в первичную поставку.
