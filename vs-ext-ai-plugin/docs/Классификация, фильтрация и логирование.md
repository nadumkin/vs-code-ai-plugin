# Классификация запросов, фильтрация отображения и логирование

Отчёт по доработке плагина AI Agent Assistant. Добавлены три связанных механизма:

1. **Определение типа запроса** — классификация пользовательского промпта (вопрос по коду, исправление ошибки, написание кода, рефакторинг, тесты и т.п.).
2. **Фильтрация отображения** — служебные данные (`<think>`-блоки, тип запроса, внутренние маркеры) не показываются в чате.
3. **Логирование** — все данные пишутся в служебные JSONL-логи для последующего анализа и улучшения.

---

## 1. Архитектура

Три новых модуля и точечная интеграция в существующий agent loop. Production-код `src/` тронут минимально и обратно совместимо (все новые зависимости опциональны — при их отсутствии работает `NullLogger` / прямой текст).

Точки интеграции:

## 2. Определение типа запроса

**Файл:** [`src/agent/RequestClassifier.js`](../src/agent/RequestClassifier.js)

### Категории

| Тип | Когда |
|---|---|
| `bug_fix` | Исправление ошибки, упавших тестов, отладка |
| `test` | Написание/прогон тестов |
| `refactor` | Рефакторинг, упрощение, переименование, оптимизация |
| `code_write` | Написание нового кода (функция, класс, модуль, эндпоинт) |
| `explain` | Объяснение/документация/обзор |
| `question` | Общий вопрос |
| `other` | Ничего не подошло |

### Как работает

Классификатор — эвристический, на наборе взвешенных правил-регулярок (двуязычных: русский + английский). Каждое правило при срабатывании добавляет вес своей категории; побеждает категория с максимальной суммой. Решение принято в пользу правил, а не отдельного LLM-вызова: это **бесплатно, мгновенно, детерминированно** и не зависит от доступности модели.

Результат:

```js
{
  type: "bug_fix",          // выбранная категория
  confidence: 0.83,         // bestScore / суммаВсехScore  ∈ [0,1]
  signals: [ {type, weight, match} ],  // какие правила сработали
  scores: { bug_fix: 10, test: 0, ... },
  promptLength: 64
}
```

Поле `confidence` — доля веса победившей категории в общей сумме; `signals` сохраняют конкретные сработавшие фрагменты (для разбора и подстройки правил по логам).

### Учёт морфологии русского

Правила покрывают разные формы: «исправь / почини / пофикс / устрани», «падает / падают / упал / упала / свалились», «напиши / создай / реализуй / сгенерируй», «рефактор / упрости / вынеси / переименуй» и т.д.

### Проверенные примеры

| Промпт | Тип | conf |
|---|---|---|
| Исправь баг — падает NullPointerException в AccountService.greet | `bug_fix` | 1.00 |
| fix the broken test for ATM withdrawal | `bug_fix` | 1.00 |
| Почему тесты падают? | `bug_fix` | 0.60 |
| Напиши класс TransactionLog с методом log(message) | `code_write` | 1.00 |
| Рефактор AccountService: вынеси валидацию | `refactor` | 0.67 |
| Напиши тесты для Cell.java | `test` | 1.00 |
| Объясни как работает метод greet | `explain` | 1.00 |
| Привет, как дела? | `question` | 1.00 |

Тип запроса — это **метаданные**: они попадают в лог (`request_classified`), но в чате не отображаются. Дальнейшее использование (адаптация промпта/набора инструментов под тип) — отдельный шаг, текущая поставка только определяет и логирует тип.

## 3. Фильтрация отображения

**Файл:** [`src/util/DisplaySanitizer.js`](../src/util/DisplaySanitizer.js)

### Принцип

Разделены три потока текста:

- **В чат (для пользователя)** — очищенный текст.
- **В лог** — сырой текст со всем содержимым (для анализа).
- **В историю для LLM** — сырой текст (модель не теряет свой контекст рассуждений).

То есть фильтрация затрагивает **только визуальный слой**; ни логи, ни диалоговая память модели не страдают.

### Что вырезается

**A. Внутри текста сообщений** (через `DisplaySanitizer`):

- Блоки рассуждений: `<think>…</think>`, `<thinking>…</thinking>`, `<reflection>…</reflection>`, `<scratchpad>…</scratchpad>`, `<internal>…</internal>` (а также `<scratch>`).
- Обрыв незакрытого блока: если ответ оборван на середине `<think>` без закрывающего тега — хвост тоже срезается.
- Строки-маркеры `[internal] …`.
- Эхо системной подсказки памяти `[Memory of past similar changes] …`.
- Метка типа запроса `[request_type=…]`.
- Тройные пустые строки, образовавшиеся после вырезания, сжимаются.

**B. Операционные сообщения (play-by-play)** — не показываются в чате по умолчанию:

После каждого инструмента ядро эмитит краткую сводку: «Прочитан файл …», «Поиск по workspace: найдено N совпадений», «Memory: сохранены стек-трейсы …», «Промежуточный ответ агента: …». Это операционный «лог», который раньше засорял чат. Теперь такие сводки уходят **только в JSONL-лог** (события `tool_result`, `assistant_message`), а в чате не появляются.

В чате остаются: богатые карточки (вывод команд, diff-ы файлов, предсказания памяти), финальный ответ ассистента, баннеры подтверждения/продолжения и ошибки.

Вернуть подробный режим (всю операционную трассу в чате) можно настройкой `aiAgentAssistant.display.showToolActivity = true`.

### Два контура защиты

1. **Хост (основной):** [`AgentRuntime`](../src/agent/AgentRuntime.js) и [`AssistantViewProvider`](../src/ui/AssistantViewProvider.js) вызывают `sanitizeAssistantText()` перед добавлением сообщения в чат; `handleToolEvent` отбрасывает внутренние system-сообщения по сигнатуре (`isLikelyInternalSystemMessage`).
2. **Webview (защитный):** [`media/main.js`](../media/main.js) повторно чистит контент при рендере (`sanitizeForDisplay`) — на случай, если что-то проскочило мимо хоста.

### Проверенные примеры

| Вход | Показано в чате |
|---|---|
| `<think>step 1\nstep 2</think>Final answer: 42` | `Final answer: 42` |
| `<scratchpad>...</scratchpad>before<reflection>x</reflection>after` | `before after` |
| `[internal] debug line\nreal answer` | `real answer` |
| `[Memory of past similar changes]\n- X failed\n\nNow the response` | `Now the response` |
| `<think>оборван без закрытия...` | *(пусто)* |

### Настройка

`aiAgentAssistant.display.hideInternalBlocks` (по умолчанию `true`).

## 4. Логирование

**Файл:** [`src/util/Logger.js`](../src/util/Logger.js)

### Формат и расположение

Один файл на время жизни окна VS Code:

```
<workspace>/.aiAgentAssistant/logs/session-<ISO-timestamp>.jsonl
```

Фолбэк при отсутствии workspace — globalStorage расширения, затем temp. Путь настраивается (`aiAgentAssistant.logging.directory`, абсолютный или относительный).

Каждая запись — одна строка JSON:

```jsonc
{ "ts": "2026-05-25T10:00:00.000Z", "seq": 12, "type": "tool_call", "payload": { ... } }
```

`seq` монотонно растёт — порядок событий восстановим даже при равных метках времени.

### Типы событий

| Событие | Когда | Ключевые поля payload |
|---|---|---|
| `session_start` | Активация расширения | version, workspace, vscodeVersion |
| `user_prompt` | Отправка запроса | prompt, fromDefault |
| `request_classified` | Классификация | type, confidence, signals, scores |
| `turn_started` | Начало турна | requestType, contextSummary, activeFile, imports, relatedTests |
| `turn_continued` | Продолжение после паузы/approval | iterationLimit, messageCount |
| `llm_request` | Запрос к модели | model, messageCount, toolCount, messages, tools |
| `llm_response` | Ответ модели | ok, status, ms, usage, choiceCount, response |
| `assistant_message` | Ответ агента на итерации | iteration, rawContent, toolCallNames, usage |
| `tool_call` | Вызов инструмента | id, name, args |
| `tool_result` | Результат инструмента | id, name, ms, ok, summary, error |
| `turn_completed` | Завершение турна | iterations, displayLength |
| `agent_final_message` | Финальный ответ в чат | rawLength, displayLength |
| `session_end` | Деактивация | — |

Этого набора достаточно, чтобы по логам восстановить полную трассу: запрос → тип → контекст → каждый обмен с моделью → каждый инструмент → финал, с таймингами и токенами (`usage`).

### Безопасность: редактирование секретов

Логгер рекурсивно проходит payload и заменяет значения ключей, совпадающих с `/(authorization|api[_-]?key|secret|token)/i`, на `[REDACTED]`. API-ключ физически не попадает в лог: он живёт в HTTP-заголовке `Authorization`, а логируется только тело запроса. Длинные строки усечены (16 КБ), глубина рекурсии ограничена.

### Надёжность

- **Не блокирует UI:** запись через асинхронную последовательную очередь (`queue = queue.then(append)`), вызывающий код не ждёт диск.
- **Сохранён порядок:** очередь сериализует записи в порядке событий.
- **Не падает:** ошибки записи проглатываются и логируются в Output-канал; `flush()` на `deactivate` никогда не бросает.
- **NullLogger:** если логгер не передан в компонент (или отключён), используется заглушка-нооп — production-путь не ломается.

### Настройки

- `aiAgentAssistant.logging.enabled` (по умолчанию `true`)
- `aiAgentAssistant.logging.directory` (пусто → `<workspace>/.aiAgentAssistant/logs/`)

## 5. Взаимодействие с бенчмарком

В headless-харнессе ([`bench/agent-driver.js`](../bench/agent-driver.js)) логирование принудительно выключено (`logging.enabled=false`): рабочие копии задач эфемерны и удаляются после прогона, JSONL там не нужен и только засорял бы диск. Классификация и санитайзинг при этом продолжают работать (они дёшевы и не имеют побочных эффектов).

## 8. Что осталось видимым в чате (а что нет)

**Видно пользователю (по умолчанию):**
- Финальный ответ ассистента (очищенный)
- Богатые карточки инструментов: вывод команд, diff-ы файлов, предсказания памяти
- Баннеры подтверждения/продолжения и ошибки

**Скрыто из чата, но сохранено в логах (и, где применимо, в истории для LLM):**
- `<think>` / `<thinking>` / `<reflection>` / `<scratchpad>` / `<internal>` блоки
- Строки `[internal] …`, эхо `[Memory of past similar changes]`, метки `[request_type=…]`
- Операционная трасса: «Прочитан файл …», «Поиск … найдено N», «Memory: …», «Промежуточный ответ агента: …» (включается через `display.showToolActivity`)

## 9. Возможные следующие шаги (за рамками текущей поставки)

- Использовать определённый тип запроса для адаптации системного промпта и набора инструментов (например, для `explain` отключать инструменты записи).
- Команда «открыть текущий лог» в палитре команд.
- Ротация/очистка старых логов по размеру или возрасту.
- Анализатор логов: агрегировать `usage`, тайминги, распределение типов запросов и долю успешных турнов для оценки качества и стоимости.
